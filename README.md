## Problem · 为什么做
深圳南山买家在售楼处只能看平面户型、听销售吹嘘，**无法客观横向比较 19 个核心豪宅**的区位 / 教育 / 医疗 / 通勤。"一线海景""顶级学区"都是话术，没人给你能验证的数字。

## Constraints · 真实世界的硬墙
- 中国大陆禁用谷歌实景三维；照片级贴图楼需付费倾斜摄影 → 免费数据天花板是 OSM 体块。
- 高德 / 天地图是 GCJ-02 坐标系，与 GPS/WGS-84 差约 500m，直接叠会错位。
- 无后端、单文件、纯前端 → 所有能力在浏览器里跑。
- 大模型会幻觉：让它直接打分，会编出不存在的配套。

## Solution · 三大能力
- 🤖 **真 AI 选房顾问**（Google Gemini 2.5 Flash）：自然语言需求 → 多约束**非线性权衡**（命中 ✓ / 让步 △ + 整体取舍）+ 经纪**带看话术**。`responseSchema` + 闭集 enum 防幻觉。
- 🏔 **3D 山海格局**（CesiumJS 1.142）：真实地形 + 天地图 / Esri 卫星 + OSM 立体楼块 + 阴影 / 暮色 / 镜头后期 + 入场导览 + 可分享视角链接。含 GCJ-02→WGS-84 火星坐标纠偏。
- 📊 **客观评分（事实层）**：教育=距最近名校、医疗=距三甲，真实坐标实测换算，每个分可溯源；事实层 / 判断层分离。对标 Walk Score 方法论本地化。

## Tradeoffs · 关键产品取舍（不是炫技）
- **为什么不让 AI 直接打分？** 会幻觉出不存在的配套 → 拆成"距离用真实坐标算 = 事实层可验证" + "AI 只解释权衡 = 判断层"，并用 schema 锁死楼盘 id 枚举，**结构性杜绝编造楼盘**。
- **为什么做非线性权衡？** 线性加权是伪精确；价值在说清"哪些能全得、哪些必须二选一"。
- **为什么楼是色块？** 照片级实景在中国要付费授权，作品集阶段不值；诚实标注"基于公开建筑数据估算"。
- **人工兜底**：用户可自己看原始距离数字，不必信 AI。

## Evaluation · 有评测的 AI 产品（不是 demo）
内置 AI 选房 Eval（控制台 `nspEval()` 或 AI 面板「🧪 跑 AI 评测」）：**20 例（16 真实买家诉求 + 4 对抗陷阱）**，用 19 盘真实坐标距离做 ground truth，对 Gemini 推荐**程序化判分**。

| 指标 | 结果 |
|---|---|
| 推荐总数 | 66 条（20 例） |
| **幻觉率（编造盘 / 错名）** | **0.0%**（66 条 0 编造） |
| **对抗陷阱通过**（不存在盘 / 不可能预算 / 矛盾诉求 / 跑题） | **4 / 4** |
| 优先级命中 | 10 / 11 |
| 平均精确率 / 召回率（对照规则基线） | 0.69 / 0.64 |
| 预算违规用例 | 1 / 20 |
| 取舍完整度 | 18 / 20 |

> 推荐用 schema 锁死 id 枚举、距离从真实坐标算（**不问 LLM**），事实层幻觉**结构性 ≈ 0**——66 条推荐零编造，可量化证明。4 个对抗陷阱全部守住：不存在的"汤臣一品深圳"不编造、500 万买深圳湾的不可能预算如实说明、"全都要最好"的矛盾诉求主动点破取舍、"哪有火锅"的跑题不硬推。

## 技术栈 / 怎么做的
高德 JS API 2.0（2D）· CesiumJS 1.142（3D）· 天地图 / Esri（卫星影像）· Turf.js（测距）· Google Gemini REST（防幻觉 schema）。单文件 HTML、零构建、CDN 加载、全程优雅降级。

**开发方式：** AI 结对编程 —— 产品 / 架构 / 数据 / 防幻觉设计与全程调试由我主导，AI 负责实现代码。

## 使用
AI 功能需访问者**自带 Gemini API key**（仅存浏览器 localStorage，不上传、不入库）。点右下「✦ AI 选房顾问」→ ⚙ 设置 key。在 https://aistudio.google.com/apikey 免费获取。

## Limitations · 诚实的局限
- OSM 楼为无贴图体块（免费数据天花板），楼高为公开估算。
- 卫星影像最高 z18（天地图深圳上限）；只覆盖南山 19 个标杆盘（架构可按片区复制）。
- AI 建议仅供参考，需实勘核实；评分不构成投资建议。高德 API 不可达时 3D 优雅降级、评分功能不受影响。
